Bygga upp en routingtopologi¶

Den grundläggande informationen för att använda de flesta av pgRouting-funktionerna id, source, target, cost, [reverse_cost] är vad som i pgRouting kallas routingtopologin.

reverse_cost är valfritt men rekommenderas starkt för att minska databasens storlek på grund av storleken på geometrikolumnerna. I den här dokumentationen används dock reverse_cost i den här dokumentationen.

När data levereras med geometrier och det inte finns någon routningstopologi behövs det här steget.

  • Var medveten om att det finns ytterligare resultat Kolumner.

  • Nya utdatakolumner är (seq, path_id, path_seq, start_vid, end_vid, node, edge, cost, agg_cost)

Observera

Standardvärde för parametern förarsida

parametern *driving side* är namnlös och giltiga värden skiljer sig åt för riktade och oriktade grafer.

  • I riktad graf: giltiga värden är [r, R, l, L]

    • Standardvärde = ”r”;

  • I en oriktad graf: giltiga värden är [b, B]

    • Standardvärde = ”b”;

Migrering av pgr_maxCardinalityMatch¶

pgr_maxCardinalityMatch works only for undirected graphs, therefore the directed flag has been removed.

Startar från v3.4.0

Signatur som ska migreras:

pgr_maxCardinalityMatch(Edges SQL, [directed])
 RETURNS SETOF (seq, edge, source, target)

Migration behövs, eftersom:

  • Använd cost och reverse_cost på den inre frågan

  • Resultaten är ordnade

  • Fungerar för oledade grafer.

  • Ny signatur

    • pgr_maxCardinalityMatch(text) returnerar endast kant kolumnen.

    • Den valfria flaggan directed tas bort.

Före migration:

  • Kolumner som används är ”utgående” och ”kommande” för att representera existensen av en kant.

  • Flaggan directed användes för att ange om det var för en directed eller undirected graf.

    • Flaggan directed ignoreras.

      • Oavsett dess värde ger det resultatet med tanke på grafen som undirected.

Migrering:

  • Använd kolumnerna cost och reverse_cost för att representera existensen av en kant.

  • Använd inte flaggan directed.

  • I frågan returneras endast kolumnen edge.

SELECT * FROM pgr_maxCardinalityMatch(
  $$SELECT id, source, target, cost, reverse_cost FROM edges$$
);

Överföring av restriktioner¶

Startar från v3.4.0

Strukturen på restriktionerna har förändrats:

Gammal begränsningsstruktur¶

På de föråldrade signaturerna:

  • Kolumnen rid ignoreras

  • via_väg

    • Måste vara i omvänd ordning.

    • Är av typen TEXT.

    • När mer än en via edge måste separeras med ,.

  • mål_id

    • Är den sista kanten av den förbjudna vägen.

    • Är av typen INTEGER.

  • till_kostnad

    • Är av typen FLOAT.

Skapande av den gamla restriktionstabellen

CREATE TABLE old_restrictions (
    rid BIGINT NOT NULL,
    to_cost FLOAT,
    target_id BIGINT,
    via_path TEXT
);
CREATE TABLE

Gamla restriktioner fylls på

INSERT INTO old_restrictions (rid, to_cost, target_id, via_path) VALUES
(1, 100,  7,  '4'),
(1, 100, 11,  '8'),
(1, 100, 10,  '7'),
(2,   4,  9,  '5, 3'),
(3, 100,  9, '16');
INSERT 0 5

Gamla restriktioner innehåll¶

SELECT * FROM old_restrictions;

Begränsningen med rid = 2 representerar \(3 \rightarrow 5 \rightarrow9`\)

  • \(3\rightarrow5\)

    • är på kolumn via_path i omvänd ordning

    • är av typen TEXT

  • \(9\)

    • är på kolumn target_id

    • är av typen INTEGER

Ny struktur för restriktioner¶

  • Kolumnen id ignoreras

  • Kolumn ”sökväg

    • Är av typen ARRAY[ANY-INTEGER].

    • Innehåller alla kanter som är involverade i begränsningen.

    • Arrayen har restriktionens ordnade kanter.

  • Kolumn kostnad

    • Är av typen ANY-NUMERICAL

Skapandet av begränsningstabellen

CREATE TABLE restrictions (
    id SERIAL PRIMARY KEY,
    path BIGINT[],
    cost FLOAT
);
CREATE TABLE

Migrering¶

Att omvandla den gamla begränsningstabellen till den nya begränsningsstrukturen,

  • Skapa en ny tabell med den nya begränsningsstrukturen.

    • I den här migreringsguiden används new_restrictions.

  • För denna migrering tillhandahåller pgRouting en hjälpfunktion för reversering av en array _pgr_array_reverse som behövs för migreringen.

    • _pgr_array_reverse:

      • Skapades temporärt för denna migration

      • Är inte dokumenterad.

      • Kommer att tas bort i nästa borgmästarversion 4.0.0

SELECT rid AS id,
  _pgr_array_reverse(
    array_prepend(target_id, string_to_array(via_path::text, ',')::BIGINT[])) AS path,
  to_cost AS cost
  INTO new_restrictions
FROM old_restrictions;
SELECT 5
