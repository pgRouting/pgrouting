Alla par - Familj av funktioner¶

Följande funktioner fungerar på alla kombinationer av toppar och par

  • pgr_floydWarshall - Floyd-Warshall’s algorithm.

  • pgr_johnson - Johnson’s algorithm

Introduktion¶

De viktigaste egenskaperna är:

  • Den returnerar inte någon sökväg.

  • Returnerar summan av kostnaderna för den kortaste vägen för varje par av noder i grafen.

  • Processen utförs endast på kanter med positiva kostnader.

  • Boost returnerar en \(V \times V\)-matris, där oändlighetsvärdena. Representerar avståndet mellan hörnpunkter för vilka det inte finns någon väg.

    • Vi returnerar endast de värden som inte är oändliga i form av en uppsättning (start_vid, end_vid, agg_cost).

  • Låt oss anta att de värden som returneras lagras i en tabell, så det unika indexet skulle vara paret: (start_vid, end_vid).

  • För den oriktade grafen är resultaten symmetriska.

    • Agg_cost för (u, v) är samma som för (v, u).

  • När start_vid = slut_vid är agg_cost = 0.

  • Rekommendation: använd en avgränsande box med högst 3500 kanter

Data¶

Följande uppgifter användes

BBOX="-122.8,45.4,-122.5,45.6"
wget --progress=dot:mega -O "sampledata.osm"      "https://www.overpass-api.de/api/xapi?*[bbox=][@meta]"

Databehandling gjordes med osm2pgrouting-alpha

createdb portland
psql -c "create extension postgis" portland
psql -c "create extension pgrouting" portland
osm2pgrouting -f sampledata.osm -d portland -s 0

Test:

Två

Detta test är med en avgränsande box Densiteten i den godkända grafen högre än i test ett. För varje <SIZE> utfördes 30 tester för att få fram genomsnittet Den testade kantfrågan är:

WITH
buffer AS (
  SELECT ST_Buffer(ST_Centroid(ST_Extent(the_geom)), SIZE) AS geom
  FROM ways),
bbox AS (
  SELECT ST_Envelope(ST_Extent(geom)) as box FROM buffer)
SELECT gid as id, source, target, cost, reverse_cost
FROM ways where the_geom && (SELECT box from bbox);

De testade frågorna

SELECT count(*) FROM pgr_floydWarshall(<edge query>)
SELECT count(*) FROM pgr_johnson(<edge query>)
